# village-emrg — чеклист ручных проверок

То, что не покрывается unit/integration/E2E и требует реального устройства,
браузера или внешнего сервиса. Заполняется по мере реализации фронта (Plan 2)
и деплоя (Plan 3).

## E2E (Playwright, Task 20)

Автоматический прогон в Chromium, поднимает реальный сервер (testcontainers PG)
и vite preview:

```
pnpm -C packages/web exec playwright test
```

Что покрыто и не требует ручной проверки:

- [x] offline-emergency: «Тревога» создаётся офлайн (текст+гео), после возврата
      сети доходит до командира. Фото в этот сценарий намеренно не входит —
      компрессия/S3 проверяются вручную (см. ниже).
- [x] moderation: командир принимает offence → инцидент становится виден
      второму жителю.
- [x] close-false: командир отклоняет emergency → исчезает из общей ленты.

E2E авторизуется без реального Telegram: test-server отдаёт login-nonce через
`/__test__/login-nonce`, фронт меняет его на сессию через `/auth/tg/exchange`.

## Telegram-бот: регистрация и вход (Plan 1/3, staging)

Бот и обмен nonce покрыты на сервере, но цепочку до реального аккаунта проверяем
руками на staging-боте:

- [ ] `/start` у боевого бота создаёт registration_request; командир видит заявку.
- [ ] После одобрения бот присылает магик-ссылку `/auth/callback?token=…`,
      открытие в браузере логинит и приземляет на ленту.
- [ ] Просроченный/повторно использованный nonce даёт экран «Не удалось войти».

## Медиа: компрессия фото (Task 07)

Канвас и Web Worker не исполняются в jsdom — проверяется вручную в браузере:

- [ ] Загрузка фото с камеры/галереи (JPEG, PNG, HEIC где поддерживается).
- [ ] Результат — WebP, длинная сторона ≤ 1920 px.
- [ ] Размер после компрессии ≤ 1 МБ для типового фото с телефона.
- [ ] EXIF (включая GPS) отсутствует в результате (canvas отбрасывает метаданные).
- [ ] До 5 фото на инцидент; шестое не добавляется.
- [ ] Компрессия идёт в Worker — UI не подвисает на больших файлах.

## Service Worker: офлайн, кэш, push (Task 09)

SW целиком исполняется только в браузере (под E2E — Task 20, и вручную):

- [ ] Первая загрузка регистрирует SW; повторный визит работает офлайн (precache app shell).
- [ ] Навигация офлайн отдаёт закэшированный HTML (NetworkFirst → cache fallback).
- [ ] Фото инцидента после первого просмотра открывается офлайн (CacheFirst media).
- [ ] Создание инцидента офлайн кладётся в IndexedDB-outbox; при возврате сети
      (событие `online` + старт авторизованной сессии) drain реплеит
      POST `/incidents`, PUT в S3 и PATCH media автоматически.
      Background Sync не используется — iOS Safari его не поддерживает.
- [ ] Web-push показывает нотификацию из payload `{title, body, url}`.
- [ ] Клик по нотификации фокусирует уже открытую вкладку инцидента либо
      открывает deep-link.

## Web Push: подписка (Task 17)

Конвертация VAPID-ключа и формирование тела подписки покрыты unit-тестами;
реальная подписка требует браузера и (на iOS) установки A2HS:

- [ ] iOS Safari: подписка работает только после «Добавить на экран» (A2HS);
      запрос разрешения вызывается по жесту пользователя, не на старте.
- [ ] `Notification.requestPermission()` → `granted` → `pushManager.subscribe`
      с боевым `VITE_VAPID_PUBLIC_KEY`; подписка уходит на `PUT /push/subscription`.
- [ ] Отказ в разрешении не шлёт запрос на сервер и не ломает UI.
- [ ] Сервер реально доставляет уведомление подписанному устройству
      (проверяется вместе с боевыми VAPID-ключами на staging).

## A2HS / манифест / установка (Task 19)

Иконки — плейсхолдеры (сплошной цвет), заменить на брендовые перед prod:

- [ ] Lighthouse → PWA installable (manifest валиден, иконки 192/512 + maskable).
- [ ] Android Chrome: `beforeinstallprompt` → кнопка «Добавить на экран»
      (видна только авторизованному), клик ставит приложение.
- [ ] iOS Safari: «Поделиться → На экран Домой» даёт standalone-иконку
      (apple-touch-icon, без адресной строки).
- [ ] В standalone-режиме status bar и тема соответствуют `theme_color`.
- [ ] Maskable-иконка корректно обрезается под круг/squircle лаунчера.
